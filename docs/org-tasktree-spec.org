#+TITLE: org-tasktree - Emacs Task Management Plugin Specification (v0.1)
#+STARTUP: showall

* 概要
org-tasktree(プラグイン名) は、org-mode を編集 UI とし、SQLite を唯一のマスターデータベースとする
Emacs 用タスク管理プラグインである。

- org バッファは常に「編集用ビュー」
- SQLite データベースが single source of truth
- DB 更新は UPDATE を使用せず、DELETE / INSERT のみで行う

* 基本コンセプト

- ノードの同一性は UID のみで保証する
- parent_id は org バッファには保持しない
- タスクは階層構造（Project / Phase / Group / Task）を持つ
- Project / Phase 未指定の Task は inbox に収容される

* DB テーブル仕様

** nodes テーブル

タスクツリーを構成するすべてのノード（Project / Phase / Group / Task）を管理する
中核テーブル。org-mode 上の見出し 1 つにつき 1 レコードが対応する。

本テーブルは delete / insert による再生成を前提とした設計とする。

*** フィールド定義

- id           INTEGER PRIMARY KEY
- uid          TEXT    NOT NULL
- parent_id    INTEGER
- node_type    TEXT    NOT NULL   ; project / phase / group / task
- todo_keyword TEXT              ; PROJ / PHASE / TODO / DONE / nil
- title        TEXT    NOT NULL
- level        INTEGER NOT NULL   ; org 見出しのレベル（* の数）
- priority     TEXT              ; "#A" / "#B" / "#C" など（org の記法をそのまま格納）
- scheduled    TEXT              ; org の SCHEDULED（ISO 日付文字列）
- deadline     TEXT              ; org の DEADLINE（ISO 日付文字列）
- closed_at    TEXT              ; DONE 時の完了日時（v0.1 では未使用）
- tags         TEXT              ; org の tag suffix（例: :tag1:tag2:）。正規化は node_tags
- status       TEXT    NOT NULL   ; OPEN / DONE
- project_id   INTEGER          ; 所属 Project の id（project ノードでは NULL）
- phase_id     INTEGER
- created_at   TEXT    NOT NULL
- updated_at   TEXT    NOT NULL

- uid はノードの同一性を保証する唯一キーであり、必須である。
- parent_id / phase_id は NULL を許容する（ルートノードなど）。
- project_id は project ノードでは NULL を許容し、それ以外のノードでは必須とする。
- todo_keyword は group ノードでは NULL になる。
- scheduled / deadline / priority / tags は未設定時 NULL を許容する。
- tags は `:tag1:tag2:` 形式で保存する（入力は `:tag1:tag2:` / `tag1:tag2` を許容し、コミット時に正規化する）。
- created_at は既存 UID の場合は維持し、新規 UID の場合は現在時刻で設定する。
- updated_at は delete / insert 再生成時に必ず現在時刻で再設定される。

*** 初期データ

- nodes テーブルには **初期化時に自動生成される inbox Project のみが初期データとして登録される。**
- inbox の id は 1 に固定とし、title = "inbox" / node_type = project / status = OPEN。
- inbox は固定 UID を持つ（システム既定値）。
- Project 未指定タスクの project_id は inbox Project の id(=1) を設定する。

---

** node_tags テーブル

ノードとタグの多対多関係を管理する正規化テーブル。
タグ検索（特に AND 条件検索）を高速化するために使用する。

本テーブルも delete / insert 再生成を前提とする。

*** フィールド定義

- node_id  INTEGER NOT NULL
- tag      TEXT    NOT NULL

- 主キー: (node_id, tag)
- 補助インデックス: (tag, node_id)
- node_id は nodes.id を参照する（外部キー制約 / ON DELETE CASCADE）。
- タグはすべて TEXT として保持し、正規化用の tags マスタテーブルは持たない。
- 1 ノードに複数タグがある場合、タグの数だけレコードを持つ。

*** 初期データ

- 初期化時に登録されるデータは **存在しない**。
- node_tags のレコードは、すべて nodes コミット時に自動生成される。
- パース規則:
  - 入力は `:tag1:tag2:` または `tag1:tag2` を許容する
  - `:` 区切りで分割し、空要素は捨てる
  - 大文字小文字は入力のまま保持する
  - 前後空白は trim する
  - 重複は除去して一意化する
  - 正規化後の nodes.tags は常に `:tag1:tag2:` 形式で保存する
  - タグ文字種の制約（例: `[A-Za-z0-9_-]+`）は v0.1 UI の入力バリデーションで行う

---

** meta テーブル

SQLite データベース自体の管理情報（スキーマバージョン、作成日時など）を管理する
キー・バリュー形式の内部管理用テーブル。

タスクデータ（nodes, node_tags）とは直接の関係を持たない。

*** フィールド定義

- key   TEXT NOT NULL
- value TEXT NOT NULL

- 主キー: key
- 本テーブルは常に key / value の 2 列構成とし、カラム追加は行わない。

*** 初期データ（初期化時に必ず INSERT される）

初期化処理（org-tasktree-init）実行時に、以下の初期データを登録する。

| key             | value                          | 説明                                     |
|------------------|----------------------------------|------------------------------------------|
| schema_version  | "1"                              | DB スキーマのメジャーバージョン         |
| created_at      | "<ISO 8601 形式の日時文字列>"   | DB 初回作成日時                          |

- created_at の例:
  - 2025-12-09T21:15:00+09:00
- schema_version は文字列として保存する。

* インデックス・制約方針

- uid 用 UNIQUE INDEX: idx_nodes_uid (uid)
- project タイトル重複禁止: idx_nodes_project_title UNIQUE (title) WHERE node_type = 'project'
- phase 重複禁止: idx_nodes_phase_title UNIQUE (project_id, title) WHERE node_type = 'phase'
- task 重複禁止: idx_nodes_task_title UNIQUE (parent_id, title) WHERE node_type = 'task'
- 検索性能向上:
  - idx_nodes_status_scheduled (status, scheduled)
  - idx_nodes_status_deadline (status, deadline)
  - idx_nodes_parent (parent_id)
  - idx_node_tags_tag (tag, node_id)
- 外部キー制約:
  - node_tags.node_id は nodes.id を参照する外部キーを付与する（ON DELETE CASCADE）。
  - parent_id / project_id / phase_id には外部キーを付与しない。
  - DB 接続時に PRAGMA foreign_keys=ON を必須とする。

* UID 運用ルール

- プラグインが展開する全ノードは必ず UID を持つ
- UID を持たないノードは「ユーザー新規作成ノード」
- コミット時に UID を必ず採番し、org バッファに即時書き戻す
- parent_id は org バッファには保持しない

* システム既定 Project（inbox）

- inbox は初期化時に自動生成され、常に 1 件のみ存在する。
- inbox の id は 1 に固定する。
- inbox は固定 UID を持つ（システム既定値）。
- node_type = project / title = "inbox" / status = OPEN。
- ユーザーからの更新・削除は不可。
- すべての同期スコープにおいて inbox は DELETE 対象から除外する。
- Project / Phase 未指定タスクは inbox に所属（project_id = inbox.id）。

* `TODO` キーワードと内部状態のマッピング仕様
** org 側 TODO キーワード

- PROJ
- PHASE
- TODO
- DONE
- nil（キーワードなし）

** 内部属性

- node_type: project / phase / group / task
- status: OPEN / DONE

** status の決定

- org-todo = DONE → status = DONE
- それ以外 → status = OPEN

** node_type の決定

- PROJ  → project
- PHASE → phase
- TODO  → task
- nil   → group
- DONE  →
  - 既存 UID がある場合: DB 上の node_type を維持
  - 新規 UID がない DONE: task とみなす

** 新規ノード（UID 未設定）かつ DONE の扱い

- UID を持たない見出しで、org-todo が DONE の場合は完全に無視する
  - UID 採番しない
  - DB に INSERT しない

* コミット仕様（org → SQLite 同期）
** コミットスコープ方式

- UPDATE は使用しない
- スコープ内 UID のみ DELETE / INSERT
- スコープ外ノードは DB 上で一切変更しない

** 親子関係の再構築

- parent_id は org ツリー構造から毎回再計算
- project_id / phase_id は最寄りの上位ノードから決定

** 繰り返しタスク（v0.1 の扱い）

- org-mode の繰り返しタスク（例: <2025-12-11 Thu +1d>）について、
  v0.1 では以下の簡易的な扱いとする。

- DB に保存する scheduled は、繰り返し指定を含まない
  「日付のみ」の値とする。
  例:
  - org:  SCHEDULED: <2025-12-11 Thu +1d>
  - DB:   scheduled = "2025-12-11"

- "+1d" などの繰り返しルール、および LAST_REPEAT や State 変更ログは
  v0.1 では DB に保存しない。
- 繰り返しタスクは、「常に現在の 1 インスタンス」としてのみ扱う。
- 過去の繰り返し履歴や、何回目の発生かといった情報は、
  v0.2 以降の「履歴管理」機能で検討する。

** delete / insert フロー

1. スコープ内 UID 収集（既存レコードの parent_id / project_id / phase_id / created_at をキャッシュ）
2. node_tags DELETE
3. nodes DELETE
4. 親ノードから順に INSERT（既存 UID の created_at はキャッシュ値を維持し、org 側で親が見つからない場合はキャッシュした既存 parent_id 等を再利用）
5. node_tags 再 INSERT
6. COMMIT

* バリデーション仕様
** nodes テーブル

*** 必須フィールド（NOT NULL チェック）

- uid
  - nil / 空文字は禁止。
  - コミット対象スコープ内で重複しないこと（論理一意性）。

- node_type
  - 許容値:
    - "project" / "phase" / "group" / "task"
  - 上記以外の値はエラー。

- title
  - nil / 空文字 / 空白のみは禁止。
  - 禁則文字: 改行（\n/\r）と制御コード（U+0000–U+001F, U+007F）


- level
  - 整数であること。
  - 1 以上であること（0 以下はエラー）。

- status
  - 許容値:
    - "OPEN" / "DONE"
  - 上記以外の値はエラー。

- created_at / updated_at
  - nil でないこと。
  - 形式は ISO 8601 形式の文字列を基本とするが、
    v0.1 では「非 NULL」のみを厳密チェック対象とする。

*** 日付関連（scheduled / deadline）

- scheduled / deadline
  - いずれも NULL を許容する。
  - DB に保存する値は `YYYY-MM-DD` 形式であること（コミット時に正規化する）。
  - UI の入力は以下を許容する（手入力・日付選択ダイアログの両方をサポートする）。
    - `YYYY-MM-DD`
    - `YYYY/MM/DD`
    - `MM-DD`（ゼロ埋めは任意）
    - `MM/DD`（ゼロ埋めは任意）
    - `DD`（1 桁も可）
  - 省略形（`MM-DD` / `MM/DD` / `DD`）は「今日を含む未来のうち最も近い日付」に解決して `YYYY-MM-DD` として保存する。
    - 例: 今日が 2025/12/15 のとき、`1` は `2026-01-01` として保存する。
  - 両方が存在する場合（いずれも非 NULL）:
    - `scheduled <= deadline` を満たすこと。

*** priority

- priority
  - NULL を許容する。
  - NULL でない場合は、英数字 1 文字であること（例: `A` / `B` / `1`）。

*** tags

- tags
  - NULL を許容する。
  - 空文字は NULL とみなす。
  - 非 NULL の場合:
    - 入力は `:` 区切り（例: `tag1:tag2`）であること。
    - 各 tag に使える文字: 半角英数字 / `-` / `_`。
  - 正規化された実体は node_tags で管理する。
  - nodes.tags は表示用キャッシュとして `:tag1:tag2:` 形式に正規化して保存してよい（org 展開時に利用する）。

*** リレーション系フィールド

- parent_id
  - NULL を許容する（ルートノードなど）。
  - NULL でない場合:
    - 対応する id を持つノードが DB 内に存在すること。

- project_id
  - project ノードでは NULL を許容する。
  - project 以外のノードでは NULL を許容しない。
  - NULL でない場合は、対応する project ノードの id であること。
  - Project 未指定の task は inbox Project の id(=1) を設定する。

- phase_id
  - NULL を許容する。
  - NULL でない場合:
    - 対応する id を持つノードが存在すること。
    - かつ、そのノードの node_type が "phase" であること。

- 外部キー制約
  - node_tags.node_id は nodes.id を参照する外部キーを付与する（ON DELETE CASCADE）。
  - parent_id / project_id / phase_id には外部キーを付与しない。
  - DB 接続時に PRAGMA foreign_keys=ON を必須とする。

*** DONE / closed_at

- closed_at
  - NULL を許容する。
  - status = "DONE" のときにのみ値を持つことが望ましい。
  - v0.1 では「NULL / 非 NULL の整合性」程度の緩いチェックに留める。

---

** node_tags テーブル

*** 必須フィールド

- node_id
  - nil は禁止。
  - 整数であること。
  - 対応する id を持つノードが nodes テーブルに存在すること。

- tag
  - nil / 空文字は禁止。
  - 前後の空白はトリムして保存すること。

*** 一意性・整合性

- (node_id, tag) の組み合わせは、1 コミット内では重複しないこと。
- node_tags の内容は、nodes.tags から再生成できる形に保つことが望ましい。

---

** meta テーブル

*** 必須フィールド

- key
  - nil / 空文字は禁止。

- value
  - nil / 空文字は禁止。

*** 初期データに関するチェック

- 初期化後の meta には少なくとも以下のキーが存在すること。

  - key = "schema_version"
    - value = "1"

  - key = "created_at"
    - value = ISO 8601 形式の文字列であることが望ましい。

*** 運用方針

- meta は内部管理用であり、通常の登録/更新系処理からは変更しない。
- 更新は初期化処理、および将来のマイグレーション処理に限定する。

* org 展開フォーマット仕様

** 基本形式
#+begin_src org-mode
 * TODO [#B] タイトル :tag1:tag2:
 SCHEDULED: <YYYY-MM-DD> DEADLINE: <YYYY-MM-DD>
 :PROPERTIES:
 :UID: xxxxx
 :END:
#+end_src

- DONE は見出しキーワードとして表示
- CLOSED 行は v0.1 では出力しない
- PROPERTIES は UID のみを保持

* エラーハンドリング方針

- 通常メッセージは `message` で出力する。
- ユーザーの操作ミスは `user-error` で通知する。
- システムエラーは `error` を使用する。
- すべてのエラーは *Messages* に記録されること。
- エラー発生時は DB 書き換えを中断し、トランザクションをロールバックする（org バッファの内容はそのまま残す）。

* コマンド仕様

本セクションでは、org-tasktree が v0.1 で提供するすべての対話コマンドについて、
目的・対象スコープ・処理内容・副作用を定義する。

** 初期化
*** org-tasktree-init

**** 目的
- SQLite データベースの初期化処理を行う。

**** 処理内容

- 以下の処理を順番に実行する。

1. `org-tasktree-database-location` で指定されたパスを取得
2. 親ディレクトリが存在しない場合は作成
3. DB ファイルが存在しない場合は新規作成
4. `nodes` / `node_tags` / `meta` テーブルを作成
5. `meta` テーブルに初期データを登録
   - schema_version = "1"
   - created_at = 現在時刻（ISO 8601）
6. システム既定 Project「inbox」を `nodes` に登録
7. `org-tasktree-query-dir` が存在しない場合は作成

**** 実行タイミング

- ユーザーが明示的に実行した場合
- 他コマンド実行時に DB が存在しない場合は自動実行

**** 副作用

- DB ファイルおよびクエリディレクトリが作成される

** 登録/更新系
*** org-tasktree-find-project
**** 目的
- project の更新/新規追加を行う。

**** 処理内容
1. nodes テーブルから の project 一覧を取得してミニバッファに表示
   - 検索条件: node_type='project' and status='OPEN'
   - ミニバッファのプロンプト表示: `find project:`
   - 表示形式: `project >` のように階層を ` > ` 区切りで表現
2. ユーザーが project を入力/選択すると project 編集バッファを開く
   - ユーザーが新規の project 名を入力した場合、新規登録として扱う。
   - ユーザーが既存の project 名を入力した場合、更新処理として扱う。
3. ユーザーが project 内容を編集し<C-c C-c> を押すとデータベースに編集内容を登録する。<C-c C-k>を押すとキャンセルして編集バッファを閉じる。

**** 編集 UI（v0.1）

- `widget.el` を使用して、表示欄と入力欄を分離した編集 UI を生成する。
- UI 上部に親 Node のパスを表示する（project の場合は空、または root 表記でもよい）。
- 入力項目:
  - title（必須）
  - priority
  - scheduled
  - deadline
  - tags
- title フィールドにはミニバッファで入力/選択された title を初期表示し、編集可能とする。
- `TAB` / `S-TAB` で入力欄を移動できる。
- `C-c C-s` で scheduled 用の `org-read-date` を起動する（`C-g` でキャンセル）。
- `C-c C-d` で deadline 用の `org-read-date` を起動する（`C-g` でキャンセル）。
- `C-c C-c` でコミットして編集バッファを閉じる。
- `C-c C-k` でキャンセルして編集バッファを閉じる。
- 各入力欄にプレースホルダーを表示し、入力フォーマットとショートカットが分かるようにする。

*重要* :
- 初期カーソル位置は title の入力欄。
- 既存ノードの UID と type は表示せず内部で保持する。

*** org-tasktree-find-phase
**** 目的
- phase の更新/新規追加を行う。

**** 処理内容
1. nodes テーブルから の project / phase 情報を取得してミニバッファに表示する。
   - 検索条件: node_type IN ('project', 'phase') and status='OPEN'
   - ミニバッファのプロンプト表示: `find phase:`
   - project をディレクトリ、phase をファイルに見立てて`find-file`と同じ操作感で phase を選択できるようにする。
   - 表示イメージ:
     - 初期表示: project 一覧を表示（`project >` 形式）
     - 特定の project を選択して TAB キーを押すと、その project 階層に移動し配下の phase 一覧を表示する。
     - phase がまだ登録されていない場合は、リストに何も表示しない。
2. ユーザーが phase を入力/選択すると phase 編集バッファを開く
   - ユーザーが新規の phase 名を入力した場合、新規登録として扱う。
   - ユーザーが既存の phase 名を入力した場合、更新処理として扱う。
3. ユーザーが phase 内容を編集し<C-c C-c> を押すとデータベースに編集内容を登録する。<C-c C-k>を押すとキャンセルして編集バッファを閉じる。

**** 編集 UI（v0.1）

- 編集 UI は `org-tasktree-find-project` と同一（`widget.el` ベース）とし、UI 上部には親 Node のパス（project）を表示する。
- title フィールドにはミニバッファで入力/選択された title（phase 名）を初期表示する。

*重要* :
- 初期カーソル位置は title の入力欄。
- 既存ノードの UID と type は表示せず内部で保持する。

*** org-tasktree-find-task
**** 目的
- task の更新/新規追加を行う。

**** 処理内容
1. nodes テーブルから の project / phase 情報を取得してミニバッファに表示する。
   - 検索条件: status='OPEN'
   - ミニバッファのプロンプト表示: `find task:`
   - project / phase / group をディレクトリ、task をファイルに見立てて`find-file`と同じ操作感で task を選択できるようにする。
   - 表示イメージ:
     - 初期表示: project 一覧を表示（`project >` 形式）
     - 特定の project を選択して TAB キーを押すと、その project 階層に移動し配下の phase / group / task 一覧を表示する。
2. ユーザーが task を入力/選択すると task 編集バッファを開く
   - ユーザーが新規の task 名を入力した場合、新規登録として扱う。
   - ユーザーが既存の task 名を入力した場合、更新処理として扱う。
   - ユーザーは新規の project/phase/group は指定できない。新規で入力した文字列は、task として扱われる。
3. ユーザーが task 内容を編集し<C-c C-c> を押すとデータベースに編集内容を登録する。<C-c C-k>を押すとキャンセルして編集バッファを閉じる。

**** 編集 UI（v0.1）

- 編集 UI は `org-tasktree-find-project` と同一（`widget.el` ベース）とし、UI 上部に親 Node のパス（project > phase > group）を表示する。
- title フィールドにはミニバッファで入力/選択された title（task 名）を初期表示する。
- scheduled / deadline は手入力と日付選択ダイアログの両方をサポートする。
  - `C-c C-s` で scheduled、`C-c C-d` で deadline の `org-read-date` を起動する。
  - `C-g` でキャンセルする。
- `C-c C-c` 実行時に `scheduled < deadline` を検査する（どちらかが未指定なら検査しない）。

*重要* :
- 初期カーソル位置は title の入力欄。
- 既存ノードの UID と type は表示せず内部で保持する。

** 検索系
*** 共通仕様
- status='OPEN' のタスクのみを対象とする。
- scheduled / deadline が NULL のタスクは除外する。
- 日付比較は `DATE(...)` で行い、日付文字列が無効な場合はエラーとして処理を中断し Messages に出力する。
- 並び順は project > phase > group > task の階層順で、同一階層内は title 昇順。

** org-tasktree-database-location

- 型: string
- 説明: SQLite データベースファイルのパス

* パフォーマンス前提

- 想定件数: 数千件規模
- スコープ付き DELETE / INSERT 方式で十分実用的

* 実装ガイドライン

- 本仕様書のみを唯一の要件定義として実装すること
- DB アクセスは Emacs 標準 sqlite API（sqlite.el）を使用する
- DB 接続直後に必ず PRAGMA foreign_keys=ON を実行する
- すべての DB 書き込みはトランザクション内で行う
- UID は UUID 形式で生成する
- UPDATE 文は禁止（DELETE / INSERT を採用）
- org-element を用いたパースを必須とする
- defcustom を用いて全設定値をユーザー変更可能にする
- ミニバッファ実装方針:
  - 本プラグインは独自のミニバッファ UI を実装しない。
  - 候補選択は必ず `completing-read`（または `read-file-name`）を使用する。
  - 補完 UI の選択（Completions, Vertico, Ivy, Fido 等）はユーザー環境に委ねる。
  - Marginalia / Vertico など外部拡張との連携は
    `with-eval-after-load` または `featurep` で存在確認を行い、
    未インストール環境では何も実行しない。
    これによりエラーを発生させない。

*** タスクツリー一括登録/更新系
**** org-tasktree-sync-buffer

***** 目的
- カレントバッファ全体を 1 つのタスクツリーとして
  SQLite データベースに登録・更新する。

***** 対象スコープ

- カレントバッファ内のすべての org 見出し

***** 処理内容

1. バッファ内のすべての heading を org-element で走査
2. 各ノードについて以下を実行
   - UID がない場合は新規採番して org バッファに即時書き戻し
   - DONE かつ UID なしのノードは無視
3. 親子関係（parent_id）を org のアウトライン構造から再構築
4. project_id / phase_id を最寄りの上位ノードから再計算
5. 対象スコープの UID をすべて収集
6. 対象 UID に対応する node_tags を DELETE
7. 対象 UID に対応する nodes を DELETE
8. 親ノードから順に nodes を INSERT
9. node_tags を再 INSERT
10. トランザクションを COMMIT

***** 副作用

- org バッファに UID が自動付与される
- DB の該当ノード群が完全に再構築される

**** org-tasktree-sync-region

***** 目的
- 選択されたリージョン内のタスクツリーのみを DB に同期する。

***** 対象スコープ

- リージョン内に含まれるすべての heading
- 各 heading のサブツリーも自動的に含める

***** 処理内容

- 処理の流れは `org-tasktree-sync-buffer` と同一とするが、
  対象範囲をリージョンに限定する。

***** 注意点

- リージョン外のノードは一切更新・削除されない
- 親ノードがスコープ外に存在する場合でも、
  parent_id は DB 上の既存構造を参照する

**** org-tasktree-sync-subtree

***** 目的
- カーソル位置の heading をルートとするサブツリーのみを DB に同期する。

***** 対象スコープ

- カーソル位置にある heading
- そのすべての子孫ノード

***** 処理内容

- 処理の流れは `org-tasktree-sync-buffer` と同一とするが、
  対象範囲をカレントサブツリーのみに限定する。

***** 適用例

#+begin_src org
 * Task A
 ** Task A-1    ← カーソル
 *** TODO A-1-1
 *** TODO A-1-2
 ** Task A-2
#+end_src

- この場合の同期対象は以下のみとする。

  - Task A
  - Task A-1
  - Task A-1-1
  - Task A-1-2
