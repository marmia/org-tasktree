#+TITLE: org-tasktree - Emacs Task Management Plugin Specification (v1)
#+AUTHOR: Masa
#+STARTUP: showall

* 概要

org-tasktree は、org-mode を編集 UI とし、SQLite を唯一のマスターデータベースとする
Emacs 用タスク管理プラグインである。

- org バッファは常に「編集用ビュー」
- SQLite データベースが single source of truth
- DB 更新は UPDATE を使用せず、DELETE / INSERT のみで行う

* 基本コンセプト

- ノードの同一性は UID のみで保証する
- parent_id は org バッファには保持しない
- タスクは階層構造（Project / Phase / Group / Task）を持つ
- Project / Phase 未指定の Task は inbox に収容される

* プラグイン名

- プラグイン名: org-tasktree

* TODO キーワードと内部状態のマッピング仕様

** org 側 TODO キーワード

- PROJ
- PHASE
- TODO
- DONE
- nil（キーワードなし）

** 内部属性

- node_type: project / phase / group / task
- status: open / done

** status の決定

- org-todo = DONE → status = done
- それ以外 → status = open

** node_type の決定

- PROJ  → project
- PHASE → phase
- TODO  → task
- nil   → group
- DONE  →
  - 既存 UID がある場合: DB 上の node_type を維持
  - 新規 UID がない DONE: task とみなす

** 新規ノード（UID 未設定）かつ DONE の扱い

- UID を持たない見出しで、org-todo が DONE の場合は完全に無視する
  - UID 採番しない
  - DB に INSERT しない

* DB テーブル仕様

** nodes テーブル

タスクツリーを構成するすべてのノード（Project / Phase / Group / Task）を管理する
中核テーブル。org-mode 上の見出し 1 つにつき 1 レコードが対応する。

本テーブルは delete / insert による再生成を前提とした設計とする。

*** フィールド定義

- id           INTEGER PRIMARY KEY
- uid          TEXT    NOT NULL
- parent_id    INTEGER
- node_type    TEXT    NOT NULL   ; project / phase / group / task
- todo_keyword TEXT              ; PROJ / PHASE / TODO / DONE / nil
- title        TEXT    NOT NULL
- level        INTEGER NOT NULL   ; org 見出しのレベル（* の数）
- priority     TEXT              ; "#A" / "#B" / "#C" など
- scheduled    TEXT              ; org の SCHEDULED（ISO 日付文字列）
- deadline     TEXT              ; org の DEADLINE（ISO 日付文字列）
- closed_at    TEXT              ; DONE 時の完了日時（v1 では未使用）
- tags         TEXT              ; 表示用キャッシュ（正規化は node_tags）
- status       TEXT    NOT NULL   ; open / done
- project_id   INTEGER
- phase_id     INTEGER
- created_at   TEXT    NOT NULL
- updated_at   TEXT    NOT NULL

- uid はノードの同一性を保証する唯一キーであり、必須である。
- parent_id / project_id / phase_id は NULL を許容する（ルートノード・ inbox など）。
- todo_keyword は group ノードでは NULL になる。
- scheduled / deadline / priority / tags は未設定時 NULL を許容する。
- created_at / updated_at は delete / insert 再生成時にも必ず再設定される。

*** 初期データ

- nodes テーブルには **初期化時に自動生成される inbox Project のみが初期データとして登録される。**
- その他のノードは、すべて org バッファのコミットによって生成される。

---

** node_tags テーブル

ノードとタグの多対多関係を管理する正規化テーブル。
タグ検索（特に AND 条件検索）を高速化するために使用する。

本テーブルも delete / insert 再生成を前提とする。

*** フィールド定義

- node_id  INTEGER NOT NULL
- tag      TEXT    NOT NULL

- 主キー: (node_id, tag)
- 補助インデックス: (tag, node_id)
- タグはすべて TEXT として保持し、正規化用の tags マスタテーブルは持たない。
- 1 ノードに複数タグがある場合、タグの数だけレコードを持つ。

*** 初期データ

- 初期化時に登録されるデータは **存在しない**。
- node_tags のレコードは、すべて nodes コミット時に自動生成される。

---

** meta テーブル

SQLite データベース自体の管理情報（スキーマバージョン、作成日時など）を管理する
キー・バリュー形式の内部管理用テーブル。

タスクデータ（nodes, node_tags）とは直接の関係を持たない。

*** フィールド定義

- key   TEXT NOT NULL
- value TEXT NOT NULL

- 主キー: key
- 本テーブルは常に key / value の 2 列構成とし、カラム追加は行わない。

*** 初期データ（初期化時に必ず INSERT される）

初期化処理（org-tasktree-init）実行時に、以下の初期データを登録する。

| key             | value                          | 説明                                     |
|------------------|----------------------------------|------------------------------------------|
| schema_version  | "1"                              | DB スキーマのメジャーバージョン         |
| created_at      | "<ISO 8601 形式の日時文字列>"   | DB 初回作成日時                          |

- created_at の例:
  - 2025-12-09T21:15:00+09:00
- schema_version は文字列として保存する。

* UID 運用ルール

- プラグインが展開する全ノードは必ず UID を持つ
- UID を持たないノードは「ユーザー新規作成ノード」
- コミット時に UID を必ず採番し、org バッファに即時書き戻す
- parent_id は org バッファには保持しない

* コミット仕様（org → SQLite 同期）

** コミット関数の種類

- org-tasktree-sync-buffer
- org-tasktree-sync-region
- org-tasktree-sync-subtree

** コミットスコープ方式

- UPDATE は使用しない
- スコープ内 UID のみ DELETE / INSERT
- スコープ外ノードは DB 上で一切変更しない

** 親子関係の再構築

- parent_id は org ツリー構造から毎回再計算
- project_id / phase_id は最寄りの上位ノードから決定

** 繰り返しタスク（v1 の扱い）

- org-mode の繰り返しタスク（例: <2025-12-11 Thu +1d>）について、
  v1 では以下の簡易的な扱いとする。

- DB に保存する scheduled は、繰り返し指定を含まない
  「日付のみ」の値とする。
  例:
  - org:  SCHEDULED: <2025-12-11 Thu +1d>
  - DB:   scheduled = "2025-12-11"

- "+1d" などの繰り返しルール、および LAST_REPEAT や State 変更ログは
  v1 では DB に保存しない。
- 繰り返しタスクは、「常に現在の 1 インスタンス」としてのみ扱う。
- 過去の繰り返し履歴や、何回目の発生かといった情報は、
  v2 以降の「履歴管理」機能で検討する。

** delete / insert フロー

1. スコープ内 UID 収集
2. node_tags DELETE
3. nodes DELETE
4. 親ノードから順に INSERT
5. node_tags 再 INSERT
6. COMMIT

* システム既定 Project（inbox）

- Project / Phase 未指定タスクは inbox に所属
- inbox は初期化時に自動生成
- 削除不可
- node_type = project
- title = "inbox"

* 検索仕様

** v1 対応

- 検索結果は必ず「専用バッファ」に出力
- カーソル位置挿入は v2 対応

** 想定検索パターン

- Project
- Project + Phase
- Schedule / Deadline
- 未スケジュール
- Priority
- Status
- Tag
- 複合条件

** ソート順（デフォルト）

1. scheduled
2. deadline
3. priority
4. title

* 検索クエリ仕様（YAML）

- クエリ定義形式は YAML のみ
- クエリファイルは org-tasktree-query-dir に保存
- Vertico で一覧選択

** YAML 例

name: mlb-today
project: MLB 調査
scheduled:
  from: 2025-12-09
  to: 2025-12-09
priority: [A, B]
status: open
tags: [mlb, dodgers]

* org 展開フォーマット仕様

** 基本形式

** TODO [#B] タイトル :tag1:tag2:
SCHEDULED: <YYYY-MM-DD> DEADLINE: <YYYY-MM-DD>
:PROPERTIES:
:UID: xxxxx
:END:

** DONE タスク

- DONE は見出しキーワードとして表示
- CLOSED 行は v1 では出力しない

** PROPERTIES

- UID のみを保持

* タスクツリー挿入時のレベル調整 (v2 対応)

- 挿入先直前の heading を L_base とする
- 挿入ルートは L_base + 1
- 検索結果側との差分 delta を全体に加算

* コマンド仕様（v1 対応）

本セクションでは、org-tasktree が v1 で提供するすべての対話コマンドについて、
目的・対象スコープ・処理内容・副作用を定義する。


** org-tasktree-init

*** 目的
- SQLite データベースの初期化処理を行う。

*** 処理内容

- 以下の処理を順番に実行する。

1. `org-tasktree-database-location` で指定されたパスを取得
2. 親ディレクトリが存在しない場合は作成
3. DB ファイルが存在しない場合は新規作成
4. `nodes` / `node_tags` / `meta` テーブルを作成
5. `meta` テーブルに初期データを登録
   - schema_version = "1"
   - created_at = 現在時刻（ISO 8601）
6. システム既定 Project「inbox」を `nodes` に登録
7. `org-tasktree-query-dir` が存在しない場合は作成

*** 実行タイミング

- ユーザーが明示的に実行した場合
- 他コマンド実行時に DB が存在しない場合は自動実行

*** 副作用

- DB ファイルおよびクエリディレクトリが作成される

** org-tasktree-sync-buffer

*** 目的
- カレントバッファ全体を 1 つのタスクツリーとして
  SQLite データベースに登録・更新する。

*** 対象スコープ

- カレントバッファ内のすべての org 見出し

*** 処理内容

1. バッファ内のすべての heading を org-element で走査
2. 各ノードについて以下を実行
   - UID がない場合は新規採番して org バッファに即時書き戻し
   - DONE かつ UID なしのノードは無視
3. 親子関係（parent_id）を org のアウトライン構造から再構築
4. project_id / phase_id を最寄りの上位ノードから再計算
5. 対象スコープの UID をすべて収集
6. 対象 UID に対応する node_tags を DELETE
7. 対象 UID に対応する nodes を DELETE
8. 親ノードから順に nodes を INSERT
9. node_tags を再 INSERT
10. トランザクションを COMMIT

*** 副作用

- org バッファに UID が自動付与される
- DB の該当ノード群が完全に再構築される

** org-tasktree-sync-region

*** 目的
- 選択されたリージョン内のタスクツリーのみを DB に同期する。

*** 対象スコープ

- リージョン内に含まれるすべての heading
- 各 heading のサブツリーも自動的に含める

*** 処理内容

- 処理の流れは `org-tasktree-sync-buffer` と同一とするが、
  対象範囲をリージョンに限定する。

*** 注意点

- リージョン外のノードは一切更新・削除されない
- 親ノードがスコープ外に存在する場合でも、
  parent_id は DB 上の既存構造を参照する

** org-tasktree-sync-subtree

*** 目的
- カーソル位置の heading をルートとするサブツリーのみを DB に同期する。

*** 対象スコープ

- カーソル位置にある heading
- そのすべての子孫ノード

*** 処理内容

- 処理の流れは `org-tasktree-sync-buffer` と同一とするが、
  対象範囲をカレントサブツリーのみに限定する。

*** 適用例

#+begin_src org
 * Task A
 ** Task A-1    ← カーソル
 *** TODO A-1-1
 *** TODO A-1-2
 ** Task A-2
#+end_src

- この場合の同期対象は以下のみとする。

  - Task A
  - Task A-1
  - Task A-1-1
  - Task A-1-2

** org-tasktree-find-query

*** 目的
- 保存済み YAML クエリーの一覧を Vertico で表示し、
  選択したクエリーをクエリー専用バッファとして開く。

*** 処理内容

1. `org-tasktree-query-dir` 配下の YAML ファイル一覧を取得
2. Vertico 補完で一覧表示
3. 選択された YAML ファイルを専用バッファとして開く

*** 副作用

- クエリー編集用バッファがカレントになる

** org-tasktree-new-query

*** 目的
- 新規クエリー作成用の専用バッファを開く。

*** 処理内容

- 空の YAML テンプレートを挿入したバッファを新規作成
- バッファは自動的に yaml-mode または text-mode で開かれる

** org-tasktree-save-query

*** 目的
- クエリー専用バッファの内容を YAML ファイルとして保存する。

*** 処理内容

1. ミニバッファでファイル名をユーザーに入力させる
2. `org-tasktree-query-dir` 配下に YAML ファイルとして保存
3. 既存ファイルが存在する場合は上書き確認を行う

** org-tasktree-search-to-buffer

*** 目的
- クエリー専用バッファの YAML を検索条件として、
  SQLite DB からタスクツリーを検索し、専用バッファに表示する。

*** 処理内容

1. クエリーバッファの YAML をパース
2. 検索条件を SQL に変換
3. SQLite DB に対して検索を実行
4. 検索結果を org 形式のタスクツリーとして生成
5. 検索結果表示専用バッファを開き、内容を挿入する

*** 表示バッファの仕様

- メジャーモード: org-mode
- 編集可能だが、直接コミットは行わない
- 編集後は通常の `org-tasktree-sync-*` 系コマンドで同期する

** org-tasktree-search-to-kill-ring

*** 目的
- クエリー専用バッファの YAML を用いて DB を検索し、
  その検索結果を org 形式の文字列として kill-ring に保存する。

*** 処理内容

1. クエリーバッファの YAML をパース
2. SQLite DB に対して検索を実行
3. 検索結果を org 形式に変換
4. 文字列として kill-ring に保存

*** 想定用途

- 任意の org バッファやメモバッファに検索結果を貼り付けたい場合

* エラーハンドリング方針

- すべてのエラーは *Messages* バッファに出力
- echo-area に短い通知を表示
- エラー発生時は DB 書き換えを中断

* カスタマイズ変数一覧

** org-tasktree-database-location

- 型: string
- 説明: SQLite データベースファイルのパス
- 初期値の決定方法:
  - 初回起動時に以下のパスが自動設定される。

    (expand-file-name "org-tasktree/tasktree.db" user-emacs-directory)

- 例:
  - ~/.emacs.d/org-tasktree/tasktree.db
  - ~/.config/emacs/org-tasktree/tasktree.db

- ユーザーが後から任意の場所に変更してもよい。

** org-tasktree-query-dir

- 型: string
- 説明: クエリー YAML ファイルの保存先ディレクトリ
- 初期値の決定方法:
  - 初回起動時に以下のパスが自動設定される。

    (expand-file-name "org-tasktree/queries/" user-emacs-directory)

- 例:
  - ~/.emacs.d/org-tasktree/queries/
  - ~/.config/emacs/org-tasktree/queries/

- ディレクトリは org-tasktree-init 実行時に自動作成される。

** org-tasktree-default-project

- 型: string
- 説明: Project / Phase 未指定タスクの所属 Project 名
- デフォルト値: "inbox"

* 初期化処理仕様

- DB ファイル作成
- テーブル作成
- meta 記録
- inbox 作成
- クエリディレクトリ作成

- org-tasktree-init で実行
- DB 不在時は自動実行

* スキーマバージョンとマイグレーション

- v1 は schema_version = 1 のみを前提
- 自動マイグレーションは v1 では非対応

* パフォーマンス前提

- 想定件数: 数千件規模
- スコープ付き DELETE / INSERT 方式で十分実用的

* 実装ガイドライン（Codex 向け）

- 本仕様書のみを唯一の要件定義として実装すること
- DB アクセスは Emacs 標準 sqlite API または emacsql-sqlite のいずれかを使用する
- すべての DB 書き込みはトランザクション内で行う
- UID は UUID 形式で生成する
- UPDATE 文は禁止
- org-element を用いたパースを必須とする
- defcustom を用いて全設定値をユーザー変更可能にする
- v2 向け機能（カーソル位置挿入等）は実装しない
