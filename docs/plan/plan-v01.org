#+TITLE: org-tasktree v0.1 実装計画
#+STARTUP: showall

* 目的 / スコープ

- org-tasktree v0.1 を仕様書（docs/org-tasktree-spec.org）に沿って実装する。
- v0.1 の対話コマンド対象:
  - org-tasktree-init
  - org-tasktree-find-project
  - org-tasktree-find-phase
  - org-tasktree-find-task
  - org-tasktree-search-today-task
  - org-tasktree-search-before-today-task
  - org-tasktree-search-overdue-task
  - org-tasktree-search-next-7day-task
- DB は SQLite（sqlite.el）を single source of truth とし、org はビュー／編集面。
- UPDATE / INSERT により更新し、既存レコードの id を維持する。
- Inbox は物理 project ノードとして常駐し、id=1 ・固定 UID を持つ。未指定 task は project_id=1。
- v0.2 以降（YAML クエリ、sync-buffer/region/subtree、履歴管理等）は本計画の対象外。

* 実装フェーズ（マイルストーン）

** M1: 基盤・ DB 初期化

- パッケージ骨格・ファイル構成を作成（org-tasktree.el を入口に分割）。
- sqlite.el 接続ヘルパ、PRAGMA foreign_keys=ON を必須化。
- スキーマ作成:
  - nodes / node_tags / meta テーブル
  - インデックス・唯一制約
  - node_tags.node_id -> nodes.id のみ外部キー（ON DELETE CASCADE）
- org-tasktree-init 実装:
  - DB/クエリ dir の自動生成
  - meta 初期データ登録
  - Inbox（id=1, 固定 UID, title="inbox"）を INSERT
  - クエリ dir 作成

受け入れ基準:
- init 実行で DB/テーブル/Inbox が一度だけ作成される。
- 再 init で Inbox が消えない。

** M2: モデル・バリデーション・共通 DB 操作

- nodes/node_tags の Elisp モデル（record/alist/struct いずれか）と変換関数。
- 必須フィールド・型・値域チェック。
- UID 生成（org-id-get-create / org-id-new）と UNIQUE 制約衝突時の再生成処理。
- commit API（uid による upsert）:
  - 既存 UID の created_at をキャッシュして維持
  - updated_at は常に現在時刻
- 検索・取得の共通 SQL ヘルパ。

受け入れ基準:
- 単体 INSERT/UPDATE が仕様通りに動き、created_at が保持される。

** M3: 検索コマンド＋org ビュー出力

- v0.1 検索共通仕様（OPEN のみ、scheduled/deadline NULL 除外、DATE 比較、並び順）を実装。
- today / before-today / overdue / next-7day の SQL。
- 結果ツリーを org-mode 形式で read-only バッファに展開し、q で閉じる。

受け入れ基準:
- サンプルデータで各検索結果と並び順が仕様通り。

** M4: UI レーン（find-project/phase/task）

- completing-read ベースの find-file 風階層ナビ。
- `widget.el` ベースの編集 UI（表示欄/入力欄の分離、C-c C-c で確定／C-c C-k でキャンセル）。
- 新規/更新の判定、DB 反映、Inbox 自動収容。

受け入れ基準:
- 単体 task をツリーを意識せず登録でき、Inbox に入る／既存編集ができる。

** M5: DB 更新方式の変更
- DELETE/INSERT 方式を廃止し UPDATE 方式に統一

受け入れ基準:
- 親 Node (project/phase)を更新しても親子関係が崩れない。

** M6: 仕上げ

- エラーハンドリング（Messages への詳細、echo-area 通知、トランザクション rollback）。
- defcustom/defgroup の整備。
- 最低限の README/仕様整合（ユースケースとコマンド一覧）。

受け入れ基準:
- 主要エラーが破綻なく通知され、DB が壊れない。

**** 補足: uid と id の役割分担の記述
#+begin_example
本プロジェクトでは、参照（parent_id / project_id / phase_id）は id を用いる。
一方 uid は「外部/永続識別子」として維持する（id 参照のままでも uid の存在理由は成立する）。

- 外部公開・永続参照:
  org 側（org-id）、URL、ログ、エクスポート、将来の連携で「一度割り当てた識別子を変えない」用途に使える。
  DB の id は移行/再投入/再構築で変わりうるが、uid は変えない前提にできる。
- データ移行・マージ耐性:
  複数 DB やバックアップのマージ時に、id は衝突するが uid は衝突しにくい。
  重複検出にも使える。
- 検索・ API の安定性:
  将来「コマンドが node を指す」時に id だと不安定、uid だと安定。

つまり、uid は「その node を“外に”指すための恒久 ID」、
id は「DB 内部の参照・最適化（JOIN 用）」という役割分担が自然である。
#+end_example
