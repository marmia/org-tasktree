#+TITLE: org-tasktree Specification - Commands
#+STARTUP: showall

* 初期化
** org-tasktree-init

*** 目的
- SQLite データベースの初期化処理を行う。

*** 処理内容
- 以下の処理を順番に実行する。

- `org-tasktree-database-location` で指定されたパスを取得
- 親ディレクトリが存在しない場合は作成
- DB ファイルが存在しない場合は新規作成
- `nodes` / `node_tags` テーブルを作成
- `PRAGMA user_version` を 2 に設定
- `org-tasktree-query-dir` が存在しない場合は作成

*** 実行タイミング

- ユーザーが明示的に実行した場合
- 他コマンド実行時に DB が存在しない場合は自動実行

*** 副作用

- DB ファイルおよびクエリディレクトリが作成される

* 登録/更新系
** org-tasktree-find-node
*** 目的
- 任意の node の更新/新規追加を行う。

*** 処理内容
1. nodes テーブルから status='OPEN' の node 情報を取得してミニバッファに表示する。
   - 検索条件: status='OPEN'
   - ミニバッファのプロンプト表示: `find node:`
   - ミニバッファのリスト表示:
     - 各 node をパス形式で表示する。(find コマンドのように'/'区切りで表示)
       - リスト表示例
         #+begin_example
         work
         work/design
         work/design/alpha
         work/design/alpha/notes
         personal
         personal/hobby
         personal/hobby/guitar
         personal/hobby/guitar/practice
         #+end_example
   - tag に domain/project/phase/group を設定している場合、tag に応じて色を分けて表示する。
     - 大小文字は区別しない。
     - tag が混在する場合は domain > project > phase > group の順で優先する。
     - Default 値:
       - domain: #40C97E
       - project: #66BFFF
       - phase: #F08AC2
       - group: #FFB366
       - task: white (domain/project/phase/group タグ無し)

2. ユーザーが node を入力/選択して'Enter'キーを押すと node 編集バッファを開く
   - ユーザーが既存の node 名を入力した場合、更新処理として扱う。
   - ユーザーが新規の node 名を入力した場合、新規登録として扱う。
     - 新規 node は既存の node パスに続けて'/<node_name>'を追記して'Enter'キーを押す。
   - 注意点:
     - 新規 node 名に'/'を含めることはできない。('/'は常に path の区切り文字として解釈される)
       - 末尾につけられた'/'は削除して編集バッファ を表示
       - node 名の途中に'/'が記述されている場合はアラート表示
     - 複数階層の新規 node を一気に作成することはできない。
       - ex:
         - 既存 node パス: project1/phase1/group1
         - 新規 node: project1/phase1/group1/task1 -> OK
         - 新規 node: project1/phase1/group1/task1/child1 -> NG (task1/child1 を一気に作成することはできない)

3. 編集 バッファの表示/入力
   - 編集バッファは `widget.el` ベースとし、UI 上部に親 node のパスを表示する。
   - 入力項目
     - title
     - priority
     - scheduled
     - deadline
     - repeat
     - tags
     - content（複数行入力）
   - title フィールドにはミニバッファで入力/選択された node パスの末端の node 名を初期表示する。
   - tags は入力欄に'/'を入力すると補完リストを表示する。TAB キーでリスト移動、Enter キーで選択可能。
   - scheduled / deadline は手入力と日付選択ダイアログの両方をサポートする。
     - `C-c C-s` で scheduled、`C-c C-d` で deadline の `org-read-date` を起動する。
     - `C-g` でキャンセルする。
   - `C-c C-c` 実行時に `scheduled < deadline` を検査する（どちらかが未指定なら検査しない）。
   - *重要* :
     - 初期カーソル位置は title の入力欄。
     - 既存ノードの UID は表示せず内部で保持する。

4. 更新/キャンセル
   - ユーザーが node 内容を編集し<C-c C-c> を押すとデータベースに編集内容を登録する。
   - <C-c C-k>を押すとキャンセルしてミニバッファ表示に戻る。

** org-tasktree-sync-buffer
*** 目的
- カレントバッファ全体を 1 つのタスクツリーとして
  SQLite データベースに登録・更新する。

*** 対象スコープ
- カレントバッファ内のすべての org 見出し

*** 処理内容
- org バッファ全体を走査し、見出し単位でノード情報を抽出する。
  - UID 未設定の見出しは自動で UID を付与する。
  - UID 付き見出しで DB に存在しない場合は user-error とする。
  - content は見出し直下の本文のみを対象とし、PROPERTIES などの drawer と SCHEDULED/DEADLINE 行は除外する。
- `DEL` フラグ付き見出しは削除対象として扱い、子孫ノードも再帰的に削除する。
  - UID 未設定の `DEL` 見出しは無視する。
- 収集したノードを `uid` をキーに INSERT/UPDATE する（id は維持）。
- parent_id は org ツリーから再計算する。
  - 親がスコープ外で特定できない場合は DB 値を保持する（keep）。

** org-tasktree-sync-region
*** 目的
- 選択されたリージョン内のタスクツリーのみを DB に同期する。

*** 対象スコープ
- リージョン内に含まれるすべての heading
- 子孫ノードはリージョン内に含まれていない限り対象外

*** 処理内容
- 選択されたリージョン内の見出しのみを対象に同期する。
  - 子孫ノードはリージョン内に含まれていない限り対象外とする。
  - parent_id の再計算と keep の扱いは `org-tasktree-sync-buffer` と同様。

** org-tasktree-sync-subtree
*** 目的
- カーソル位置の heading をルートとするサブツリーのみを DB に同期する。

*** 対象スコープ
- カーソル位置にある heading
- そのすべての子孫ノード

*** 処理内容
- カーソル位置の見出し配下のタスクツリーを対象に同期する。
  - 子孫ノードも更新対象に含める。
  - parent_id の再計算と keep の扱いは `org-tasktree-sync-buffer` と同様。

*** 適用例

#+begin_src org
 * Task A
 ** Task A-1    ← カーソル
 *** TODO A-1-1
 *** TODO A-1-2
 ** Task A-2
#+end_src

- この場合の同期対象は以下のみとする。

  - Task A
  - Task A-1
  - Task A-1-1
  - Task A-1-2

* 検索系
** 共通仕様
- status='OPEN' の node のみを対象とする。
- 日付比較は `DATE(...)` で行い、日付文字列が無効な場合はエラーとして処理を中断し Messages に出力する。
- 並び順は親子順で、同一階層内は title 昇順。
- repeat!=NULL の場合は、tasktree 展開時に'<SCHEDULE yyyy-mm-dd>'に`repeat`に設定されている値を付け足して出力する。
  - Ex: <SCHEDULED 2025-12-20 .+1d>
- content が NULL でない場合は、見出し/日付/PROPERTIES の直後に本文として出力する（余分な空行は挿入しない）。

** org-tasktree-search-today-task
*** 目的
- scheduled が今日になっている node のツリーを取得する。
*** 処理内容
- nodes テーブルから`scheduled=today and status=OPEN`の node と、その node の祖先 node を取得する。
  - node 情報は重複なく正規化したデータを取得
  - 所得する属性情報: uid, todo_keyword, title, priority, scheduled, deadline, tags, content
- 出力バッファを開き、org-mode 形式でツリーを展開する
  - 出力バッファは`q`キーで閉じる

** org-tasktree-search-by-query
*** 目的
- `org-tasktree-query-dir`配下の YAML クエリーで tasktree を検索・取得する。
*** 処理内容
1. `org-tasktree-query-dir`配下の `.yml`/`.yaml` をミニバッファに表示し、選択または新規ファイル名入力でクエリー編集バッファを開く。
2. クエリー編集バッファは `yaml-mode` を基本とし、`C-c C-c`で検索実行、`C-c C-k`でキャンセルして閉じる。
3. クエリーフォーマットが不正な場合は `user-error` を出力する。
4. 検索結果を出力バッファに org-mode 形式で表示する。
   - 出力バッファは `q` キーで閉じる。
*** クエリーフォーマット
#+begin_src yaml
todo_keyword:
title:
priority:
scheduled:
deadline:
repeat:
closed_at:
tags:
content:
status:
created_at:
updated_at:
include_ancestor:
include_descendants:
#+end_src

- `field_name:` に続けて検索条件を指定する。
- 検索条件の前後の空白は無視する。
- 固定値は完全一致検索。
- 複数項目を指定した場合は AND 検索。
- 1 項目内に `and` / `or` を指定可能（大文字可）。
  - `and` は `or` より優先して評価する。
  - 括弧は扱わない。
- 未クォートの空白区切りは AND として扱う。
  - 例: `title: foo bar` → `foo AND bar`
- 空白を含めた文字列を検索したい場合は、ダブルクオートまたはシングルクォートで囲む。
  - 例: `title: "Hello World"` / `title: 'We are the world.'`
- 文字列として `and` / `or` / `not` を使いたい場合はクォートで囲む。
  - 例: `title: "and"`
- クォート内では `\"` / `\'` / `\\` をエスケープとして扱う。
- 比較記号（`<` `>` `=` `<=` `>=` `<>`）と `not` キーワードは検索条件の先頭に付ける。
  - 例: `scheduled: <= 2026-01-10`
  - 例: `tags: not NULL`
- `NULL` で検索したい場合は `NULL` を記述する。
  - 文字列 "NULL" を検索したい場合は `\NULL` と記述する。
- ワイルドカード `*` `?` を使用できる。
  - `*` は任意長、`?` は 1 文字として扱う。
  - `\*` `\?` はリテラルとして扱う。
  - ワイルドカード検索は大文字小文字を区別しない。
- 日付に指定可能な形式は `yyyy-MM-dd` または `yyyy/MM/dd`。
  - `today`/`yesterday`/`tomorrow`（大文字小文字可）を指定できる。
  - `today+7d` / `today +7d` のように日数を加減算できる。
  - 上記以外の形式は `user-error` とする。
- `tags` 検索は `nodes.tags` に対して検索を行う。
  - クエリー指定はスペース区切り。
  - 指定した tag を **全て含む** node を対象とする。
    - 例: `aa bb` → `aa:bb:cc` は対象、`aa:cc`/`bb:cc` は対象外。
- `include_ancestor` / `include_descendants` は true/false（大文字可）。
  - 未指定時の既定値は両方 true。
- `include_ancestor=true` の場合、該当 node の祖先を含める。
- `include_descendants=true` の場合、該当 node の全子孫を含める。
- 両方 false の場合、該当 node のみを出力する。
  - 出力は「検索結果に含まれる node のみ」でツリーを再構成する。
  - 親が含まれていない node は root として出力する。
- バリデーションは以下のみ実施する。
  - scheduled / deadline / created_at / updated_at
  - include_ancestor / include_descendants
- クエリーファイル新規作成時は、バッファ内に全キー項目をセットする。
  - 値は、`include_ancestor`, `include_descendants` に`true`を設定し、他のキー項目の値は空値にする。
- 未指定/未使用/未定義の項目は無視する。
- SQL インジェクションに対して最低限の対策を実施する。
