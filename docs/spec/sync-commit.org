#+TITLE: org-tasktree Specification - Sync and Commit
#+STARTUP: showall

* `TODO` キーワードと内部状態のマッピング仕様
** org 側 TODO キーワード

- PROJ
- PHASE
- TODO
- DONE
- nil（キーワードなし）

** 内部属性

- node_type: project / phase / group / task
- status: OPEN / DONE

** status の決定

- org-todo = DONE → status = DONE
- それ以外 → status = OPEN

** node_type の決定

- PROJ  → project
- PHASE → phase
- TODO  → task
- nil   → group
- DONE  →
  - 既存 UID がある場合: DB 上の node_type を維持
  - 新規 UID がない DONE: task とみなす

** 新規ノード（UID 未設定）かつ DONE の扱い

- UID を持たない見出しで、org-todo が DONE の場合は完全に無視する
  - UID 採番しない
  - DB に INSERT しない

* コミット仕様（org → SQLite 同期）
** コミットスコープ方式

- 対象スコープ内のノードは `uid` をキーに INSERT/UPDATE する（nodes の DELETE/INSERT 再生成は行わない）。
- node_tags は node_id 単位で DELETE / INSERT により再生成してよい。
- スコープ外ノードは DB 上で一切変更しない

** create/update の境界

- `uid` が DB に存在する場合は update（UPDATE）として扱う。
- `uid` が DB に存在しない場合は create（INSERT）として扱う。

** UPDATE 可能列 / 禁止列（nodes）

- 禁止（不変）:
  - id
  - uid
  - node_type
  - created_at
- 更新可能:
  - parent_id / project_id / phase_id
  - todo_keyword / level / title
  - priority / scheduled / deadline / tags
  - status / closed_at
  - updated_at（UPDATE のたびに現在時刻）

** 親子関係の再構築

- parent_id は org ツリー構造から毎回再計算
- project_id / phase_id は最寄りの上位ノードから決定

** 繰り返しタスクの扱い

- DB に保存する scheduled は、繰り返し指定を含まない「日付のみ」の値とする。
  例:
  - org:  SCHEDULED: <2025-12-11 Thu +1d>
  - DB:   scheduled = "2025-12-11"

- "+1d" などの繰り返しルール、および LAST_REPEAT や State 変更ログは DB に保存しない。
- 繰り返しタスクは、「常に現在の 1 インスタンス」としてのみ扱う。

** insert / update フロー

1. スコープ内ノードを収集（既存レコードの id / node_type / parent_id / project_id / phase_id / created_at をキャッシュ）
2. 親子関係（parent_id / project_id / phase_id）を org ツリーから再計算（親がスコープ外で org 側から特定できない場合はキャッシュ値を維持）
3. トランザクション開始
4. 親ノードから順に、`uid` が既存の場合は UPDATE、未登録の場合は INSERT（id/uid/node_type/created_at は不変、updated_at は現在時刻）
5. node_tags を対象 node_id について DELETE し、正規化後のタグを INSERT
6. COMMIT
